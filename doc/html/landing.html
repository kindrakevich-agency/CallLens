Архітектура

Шлях дзвінка крізь систему

П'ять зон обробки. Синхронне приймання на Symfony повертає відповідь за мілісекунди, а вся «важка» робота йде асинхронно у воркерах і викликає зовнішні API . Власний GPU не потрібен.

  1. A
    Зона · Клієнт
    поза вашим контуром

    Джерело дзвінка

    • Джерело Twilio · Binotel · Ringostat · Bitrix24
    • Подія завершення дзвінка
    • Дані запис mp3/wav, бажано двоканальний (менеджер/клієнт окремо)
  2. B
    Зона · Symfony (синхронно)
    < 50 мс

    Приймання вебхука

    • Endpoint POST /v1/calls/webhook
    • Безпека перевірка підпису HMAC-SHA256
    • Дедуп ідемпотентність за call_id
    • Дія зберегти аудіо в Hetzner S3 → dispatch у Messenger
    • Відповідь 202 Accepted одразу, не чекаючи обробки
  3. C
    Зона · Воркери (асинхронно)
    Messenger

    Обробка дзвінка

    • Транспорт Redis / Doctrine · ретраї з backoff · dead-letter
    • Виклики через Symfony HttpClient за єдиним інтерфейсом
    Конвеєр · зовнішні API
    платно/хв
    1 · STTDeepgram / AssemblyAI → транскрипт + діаризація + таймкоди слів
    2 · LLMGPT-4o-mini / Gemini → JSON-оцінка по критеріях + цитати (structured output)
    3 · EMBOpenAI / Voyage → вектори реплік для пошуку
  4. D
    Зона · Дані
    Doctrine ORM

    PostgreSQL + pgvector

    • Сутності Call · Transcript · Utterance(+vector) · Score · Tenant
    • Ізоляція дані розділені за tenant_id (мультитенантність)
    • Пошук семантичний по векторах (pgvector, індекс HNSW)
  5. E
    Зона · Інтерфейс
    REST

    Подача та дашборд

    • API Symfony REST (JSON), автентифікація за токеном і тенантом
    • Аналітика Cube — семантичний шар і пре-агрегації над PostgreSQL для швидких звітів
    • UI Next.js (React) — оцінки, транскрипт, звіти, пошук по дзвінках

Бурштинова зона C — єдина, що залежить від зовнішніх платних API. Усе сховано за інтерфейсом, тож провайдера можна замінити, а згодом — перенести транскрипцію на власний GPU, не чіпаючи зони A, B, D, E.

Рішення про інфраструктуру

Чому API, а не власний GPU

На старті й ранньому зростанні платні API дешевші за власний GPU у 3–4 рази — і значно простіші в експлуатації. Власний GPU стає вигідним лише на великому обсязі.

Метрики поверх PostgreSQL з Cube

Cube — семантичний шар над базою: метрики описуються один раз, а віддаються через API з пре-агрегаціями та кешем. Звіти лишаються швидкими на самому PostgreSQL — без окремого пошукового рушія.

cubes:
  - name: calls
    sql_table: calls
    measures:
      - name: avg_score
        sql: score
        type: avg
      - name: count
        type: count
    dimensions:
      - name: manager
        sql: manager_id
        type: string
      - name: week
        sql: created_at
        type: time
    pre_aggregations:
      - name: by_manager_week
        measures: [avg_score, count]
        dimensions: [manager]
        granularity: week

Підключення джерела

Що таке «реєстрація вебхука»

Після реєстрації в розділі Налаштування → Інтеграції ви отримуєте власну вебхук-адресу та секретний ключ. Цю адресу ви вказуєте у своїй телефонії чи CRM.

Коли дзвінок завершується, ваша система надсилає на цю адресу повідомлення із посиланням на запис. Symfony перевіряє підпис, зберігає аудіо й ставить дзвінок у чергу на обробку. Одне налаштування на старті — далі автоматично.

  • ✓Підпис HMAC-SHA256 захищає від підробних запитів.
  • ✓Двоканальні записи — менеджер і клієнт на окремих доріжках, без помилок діаризації.
  • ✓Немає вебхука — записи можна завантажувати вручну або через REST API.
