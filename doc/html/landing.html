Architecture

A call's journey through the system

Five processing zones. Synchronous intake on Symfony returns a response in milliseconds, while all the heavy lifting runs asynchronously in workers that call external APIs . No GPU of your own required.

  1. A
    Zone · Customer
    outside your perimeter

    Call source

    • Source Twilio · Binotel · Ringostat · Bitrix24
    • Event call completed
    • Data mp3/wav recording, ideally dual-channel (rep/customer on separate tracks)
  2. B
    Zone · Symfony (synchronous)
    < 50 ms

    Webhook intake

    • Endpoint POST /v1/calls/webhook
    • Security HMAC-SHA256 signature verification
    • Dedupe idempotency by call_id
    • Action store audio in Hetzner S3 → dispatch to Messenger
    • Response 202 Accepted immediately, without waiting for processing
  3. C
    Zone · Workers (asynchronous)
    Messenger

    Call processing

    • Transport Redis / Doctrine · retries with backoff · dead-letter
    • Calls via Symfony HttpClient behind a single interface
    Pipeline · external APIs
    paid/min
    1 · STTDeepgram / AssemblyAI → transcript + diarization + word timestamps
    2 · LLMGPT-4o-mini / Gemini → JSON scoring per criterion + quotes (structured output)
    3 · EMBOpenAI / Voyage → utterance vectors for search
  4. D
    Zone · Data
    Doctrine ORM

    PostgreSQL + pgvector

    • Entities Call · Transcript · Utterance(+vector) · Score · Tenant
    • Isolation data partitioned by tenant_id (multi-tenancy)
    • Search semantic vector search (pgvector, HNSW index)
  5. E
    Zone · Interface
    REST

    Delivery and dashboard

    • API Symfony REST (JSON), authenticated by token and tenant
    • Analytics Cube — a semantic layer with pre-aggregations over PostgreSQL for fast reports
    • UI Next.js (React) — scores, transcript, reports, call search

The amber zone C is the only one that depends on external paid APIs. Everything sits behind an interface, so you can swap providers and, down the line, move transcription onto your own GPU without touching zones A, B, D, or E.

Connecting a source

What "registering a webhook" means

After signing up, head to Settings → Integrations and you'll get your own webhook URL and secret key. You paste this URL into your phone system or CRM.

When a call ends, your system sends a message to that URL with a link to the recording. Symfony verifies the signature, stores the audio, and queues the call for processing. One setup at the start — automatic from there on.

  • ✓An HMAC-SHA256 signature guards against forged requests.
  • ✓Dual-channel recordings put the rep and customer on separate tracks, with no diarization errors.
  • ✓No webhook? You can upload recordings manually or via the REST API.

public function handle(Request $r): Response
{
    $this->verifySignature($r);        // HMAC
    $bus->dispatch(new ProcessCall($payload));
    return new Response(status: 202);
}

Infrastructure · hardware

One dedicated server is enough

All the heavy processing — transcription, scoring, embeddings — runs on external APIs, so the server needs no GPU. It only hosts the app, the workers, the database, and the queue — comfortably handled by a single Hetzner dedicated server.
